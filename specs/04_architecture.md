# 04 — Architecture

## System diagram

```
┌────────────────┐         ┌─────────────────┐
│  Next.js 14    │  HTTP   │  FastAPI        │
│  (frontend)    │ ──────▶ │  (backend)      │
│  :3000         │   JSON  │  :8000          │
└────────────────┘         └────────┬────────┘
                                    │
                  ┌─────────────────┼──────────────────┐
                  ▼                 ▼                  ▼
          ┌──────────────┐  ┌──────────────┐   ┌──────────────┐
          │  Postgres 15 │  │  APScheduler │   │  Anthropic   │
          │  :5432       │  │  (in-proc)   │   │  API         │
          └──────────────┘  └──────┬───────┘   └──────────────┘
                                   │
                                   ▼
                            ┌──────────────┐
                            │  GitHub API  │
                            │  REST + GQL  │
                            └──────────────┘
```

Four services in Docker Compose: `postgres`, `backend`, `frontend`, `scheduler`. The `scheduler` is just a second backend container started with `python -m repolens.scheduler` instead of uvicorn — same image, different entrypoint. Keeps deploy simple.

## Backend (FastAPI)

### Layout
```

├── app/
│   ├── main.py                  # FastAPI app, mounts routers, CORS, auth dep
│   ├── config.py                # pydantic-settings, env vars
│   ├── db.py                    # SQLAlchemy engine + session
│   ├── models/                  # SQLAlchemy ORM models
│   ├── schemas/                 # Pydantic request/response models
│   ├── routers/
│   │   ├── auth.py              # OAuth callback, /me
│   │   ├── repos.py             # /repos, /repos/{id}
│   │   ├── inbox.py             # /inbox
│   │   ├── triage.py            # /triage
│   │   ├── releases.py          # /releases
│   │   ├── digest.py            # /digest, POST /digest/generate
│   │   └── settings.py          # /settings, PAT/anthropic key updates
│   ├── services/
│   │   ├── github_client.py     # thin wrapper over httpx + retry/backoff
│   │   ├── sync.py              # orchestrates a sync run
│   │   ├── inbox_builder.py     # rebuilds inbox_items table
│   │   ├── priority.py          # priority_score function
│   │   ├── digest_generator.py  # Claude prompt + response parser
│   │   ├── release_notes.py     # categorize PRs, draft notes
│   │   └── crypto.py            # AES-GCM for PAT/API key
│   └── scheduler.py             # APScheduler entrypoint, job definitions
├── alembic/                     # migrations
├── tests/
└── pyproject.toml
```

### Why FastAPI
Already proven across customer360, MetricAnchor, sigmaops. Type-safe via pydantic, async-first (matters for GitHub fan-out). Auto-OpenAPI docs at `/docs` for free.

### Why APScheduler (not Celery, not cron)
- Single-binary feel — no Redis/RabbitMQ broker
- Persists job state to Postgres (we already have it)
- Sufficient for v0.1 cadence (30-min sync, weekly digest)
- Swap to Celery later if multi-worker needed

### Auth
- v0.1: GitHub OAuth → server-side session cookie (HttpOnly, SameSite=Lax)
- Single user mode: app refuses second registration. Error: *"RepoLens v0.1 is single-user. Sign out the existing user first or run a second instance."*
- PAT generated by user (legacy approach) OR exchanged via OAuth — support both

### Encryption
- PAT and Anthropic key encrypted at rest with AES-GCM
- Master key from `REPOLENS_ENCRYPTION_KEY` env var (32-byte hex)
- Decrypted only in-memory, never logged. Crypto layer in `services/crypto.py` is the only place that touches the key

### Rate limiting (GitHub)
- Track `X-RateLimit-Remaining` header per response, store in `sync_runs`
- If remaining < 100, pause sync, log a banner that the UI can show
- ETag-aware GETs to avoid burning quota on unchanged data
- Exponential backoff on 403/secondary rate limits

### Why this stack
- App Router for nested layouts (sidebar persists, content swaps)
- Server Components for repo wall (fetch on server, stream HTML)
- Client Components for interactive Inbox (hotkeys, optimistic updates)
- shadcn/ui keeps design coherent without dragging in a heavy framework

### Data fetching
- Server: direct fetch to FastAPI from RSC (`fetch(API_URL/...)`)
- Client: SWR for live data + revalidation on focus
- Cache: 30s for lists, 5m for repo metadata, no-cache for digest generation

## Deployment

### Docker Compose (the canonical way to run RepoLens)
```yaml
services:
  postgres:        # 15-alpine
  backend:         # uvicorn, port 8000
  scheduler:       # python -m repolens.scheduler (same image as backend)
  frontend:        # next start, port 3000
```

Single-command spin-up:
```
docker compose up -d
```

First-run wizard handles DB migrations (Alembic) and OAuth setup.

### Self-host story
- Hetzner / DO / Pi at home — all fine
- ~ 200 MB RAM idle; ~500 MB when syncing
- Postgres volume is the only stateful thing; document `pg_dump` for backup
- Optional: provide a one-click Render / Fly.io / Coolify recipe

## Observability (lightweight)

- Structured logs via `structlog` — JSON to stdout
- `/healthz` endpoint hits DB
- `/metrics` endpoint Prometheus-format (sync count, API call count, rate-limit remaining gauge)
- No external telemetry by default — if user wants it, they wire it up

## Security posture

- All cookies HttpOnly + SameSite=Lax + Secure (when HTTPS detected)
- CSRF: SameSite=Lax + custom header check on mutations
- All API routes require auth except `/healthz` and OAuth callback
- Encryption keys via env, never in DB; rotation = re-encrypt on next request
- Dependabot configured on the repo itself (eat your own dog food)

## What we deliberately don't build

- No Kubernetes manifests — Compose is the supported path
- No multi-tenant isolation — single-user app, even if schema supports more
- No GraphQL gateway — REST is fine for this app's surface
- No background queue beyond APScheduler — only add Celery if a real use case demands it
